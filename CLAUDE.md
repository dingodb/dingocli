# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`dingo` is a Go CLI for deploying and managing DingoFS / dingo-store / dingodb clusters. Despite the repo name `dingofs-tools`, the Go module path is `github.com/dingodb/dingocli` and the produced binary is `bin/dingo` (entry point `cmd/dingo/main.go`).

## Common commands

```sh
# Build (requires musl-gcc; uses CGO + static linking for the embedded sqlite driver)
make build              # full release build (runs `make proto` first)
make debug              # build with -N -l for delve

# Tests
make test                       # run all tests
make test CASE=TestFooBar       # run a single test by regex (passed to `go test -run`)

# Lint (installs pinned golangci-lint v1.50.0 into $GOPATH/bin)
make lint

# Protobufs — regenerate Go stubs from the third-party/dingofs-proto submodule
make proto              # invokes mk-proto.sh; needs the submodule initialized
git submodule update --init --recursive

make clean              # removes bin/ AND proto/ (regenerated by `make proto`)
```

Notes:
- The build toolchain hard-codes `CC=musl-gcc` and `CGO_LDFLAGS="-static"`. On hosts without musl-gcc, override with `make build CC=gcc CGO_LDFLAGS=""` or build the dev target.
- `proto/` is generated output — do not edit by hand. Source `.proto` lives in `third-party/dingofs-proto` (git submodule).
- `vendor/` is checked in; Go modules resolve from there.
- Version, commit, and build date are injected via `-ldflags -X github.com/dingodb/dingocli/cli/cli.*`. Bump the version literal in the `Makefile`, not in source.

## Architecture

### Command tree (cobra)

`cli/dingocli.go` builds a `DingoCli` runtime, then `cli/command/cmd.go::NewDingoCliCommand` wires the cobra tree. Top-level subcommands are grouped into `ADMIN` / `DEPLOY` / `UTILS` and live in `cli/command/<name>/`:

- `cluster`, `hosts`, `config`, `playbook`, `ssh`, `enter`, `exec` — deployment/lifecycle
- `fs`, `mds`, `cache`, `nfs`, `monitor`, `component` — service/feature management
- `audit`, `completion` — utilities

Each subcommand package owns a `cmd.go` that registers its children. Mirror that pattern when adding commands rather than dumping new files into `cli/command/`.

### Execution model: Playbook → Tasks → Task → Steps

Cluster operations are not direct shell calls; they go through a four-level pipeline (see `internal/playbook/playbook.go` for the canonical diagram):

```
Playbook
└── Tasks (one type, e.g. "create container")
    └── Task (a single host or role)
        └── Step (a single action: shell, file copy, container op, etc.)
```

- `internal/task/step/` — primitive steps (`shell.go`, `file.go`, `container.go`, plus checker helpers)
- `internal/task/task/common/` and `internal/task/task/monitor/` — composed Task constructors (pull image, sync config, start service, etc.) used by cluster vs. monitor flows respectively
- `internal/task/task/checker/` — precheck tasks (kernel, network, ssh, permissions, topology validation)
- `internal/tasks/` and `internal/playbook/tasks/` — concurrent task runners with progress bars (mpb) and a shared `monitor` for OK/SKIP/ERROR status
- `internal/playbook/factory.go` — builds the right Tasks for a given `PlaybookStep.Type`

When adding a new cluster operation, you almost always: (1) define step constants and a Task constructor under `internal/task/task/...`, (2) register it in `internal/playbook/factory.go`, (3) push a `PlaybookStep` from the cobra command.

### Configuration / topology

- `internal/configure/topology/` parses the user's cluster YAML (kinds: `dingofs`, `dingodb`, `dingo-store`) into `DeployConfig` structs. Services are matched by role (`ROLE_MDS`, `ROLE_METASERVER`, etc.); the role set is kind-dependent — see `CheckRole` in `cli/cli/cli.go`.
- `internal/configure/hosts/` parses the hosts file; `internal/configure/dingocli/` parses `~/.dingo/dingocli.cfg`.
- Sample topologies for users live under `configs/fs/` and `configs/store/`.

### Runtime state

`NewDingoCli` provisions `~/.dingo/{bin,data,plugins,logs,temp}` and opens a SQLite or rqlite store (`internal/storage/`) keyed by cluster. Clusters, hosts, services, audit logs, and version info all persist there — most "get the current X" helpers on `DingoCli` read from this storage, not from process state.

The current cluster can be overridden per-invocation via the env var checked in `getActivatedClusterFromEnv` (`internal/common`).

### Auditing

Every non-`audit`/`__complete` invocation is bracketed by `dingocli.PreAudit` / `PostAudit` in `cli/dingocli.go`, writing the command, cwd, and final status (success/fail/cancel + error code) to storage. Cobra command errors must return `*errno.ErrorCode` (`internal/errno`) to record a structured code.

### RPC clients

`internal/rpc/` wraps the generated gRPC stubs (mds, cachegroup, etc.) for commands like `dingo fs list` that talk to a live cluster instead of orchestrating one.

## Conventions worth knowing

- Errors flow through `internal/errno`. New error conditions get a code there; commands return the code, and the audit layer extracts it.
- User-facing output goes through `dingocli.Out()` / `WriteOut` / `WriteOutln`, not `fmt.Println`, so it can be redirected in tests.
- Progress UI uses `vbauerster/mpb` via the Tasks runner — don't print directly during a Tasks run or you'll corrupt the bars.
- Tests use `-p 3 -count=1`; the `CASE` variable feeds `-run`, so it's a regex (e.g. `CASE='TestParse.*'`).
